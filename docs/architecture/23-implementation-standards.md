# KEEL — Implementation Standards

> Document 23 · Status: FROZEN — Architecture v1.0 (2026-07-12)
> Project-wide conventions. Where a convention repeats a Constitution law (Doc 22), the Constitution number is cited; the Constitution wins on conflict.

| Area | Standard |
|------|----------|
| **Folders** | One folder per module exactly as in Doc 03 §2; module-internal layering is flat (no `utils/` dumping grounds inside modules — a file that doesn't belong is a design question, not a folder question). Tests colocated in `__tests__/` next to the code; black-box suites under root `tests/`. |
| **Naming** | Files: kebab-case. Types/entities: PascalCase nouns from the ubiquitous language (Doc 01 §1) — synonyms are forbidden vocabulary (no "Recording" for Snapshot, no "Diff" for Divergence). Ports: capability nouns (`Runner`, `ObjectStore`, `Logger`) — never `IRunner`, never `RunnerImpl`; implementations are named by what makes them concrete (`OllamaProvider`, `SqliteBaselineRepository`). Booleans read as predicates; functions are verb-first; constants SCREAMING_SNAKE only for true constants. |
| **Interfaces (ports)** | Defined in the consumer module (C22); one file per port; documented with: purpose, invariants, error contract, cancellation budget. Every port has a contract-test kit (C67) and at most the implementations the roadmap requires (C29). |
| **DTOs** | Adapters exchange DTOs, not entities; DTOs are versioned with the surface they belong to (MCP schema, CLI output). Mapping lives in the adapter, is total (no partial projections that drop fields silently), and is golden-tested. Internal engines pass Behavior Model types, never DTOs. |
| **Errors** | Typed hierarchy per Doc 10 C1 (C58–61). Every thrown error includes: stable machine code (`KEEL_E_*`), human message meeting the what/why/fix bar, and structured context fields — never string-interpolated context. Wrapping preserves cause chains. |
| **Logging** | Injected Logger port only (C62); event names are `module.action.outcome` (e.g. `capture.seal.rejected`); every line carries `opId` (C63); levels per Doc 10 B; no logging in `model/` or `diff/`. |
| **Configuration** | All access via ConfigSnapshot (C64); adding a key requires: schema entry + default + behavior/presentation classification + generated-doc description, in the same PR. |
| **Testing** | Vitest; property tests via fast-check for every algebraic law claimed in Doc 13; no mocking frameworks for domain types — hand-rolled fakes implementing the port contract; test names state the behavior, not the method (`rejects nondeterministic probe naming the flapping path`). Coverage is not a gate; the Doc 25 test gates are. |
| **Documentation** | Every module has a `README.md` stating: contract reference (Doc 20 section), what may import it, extension points. Reference docs (config keys, verdict format, MCP tools, error codes) are generated from schemas — hand-written reference is forbidden (drift). Guides are task-oriented. |
| **Imports** | ESM only; module-boundary imports only via each module's public index (deep imports across modules fail lint); import order: node builtins → external → `@keel/*` → module-relative; no path aliases that hide layering. |
| **Dependency injection** | Constructor injection, wiring only in the two composition roots (C27, C30); constructors take a single dependencies object; nothing constructs its own dependencies except pure values. |
| **Repositories** | Per-consumer ports (Doc 20 §8); methods express domain operations (`sealBaseline`), never CRUD verbs (`update`); all methods transactional at their stated grain; SQL only inside `storage/` (C36). |
| **Code review** | Every PR: one approving maintainer review; boundary-affecting PRs cite the Doc 20/21 contract they operate under; golden-file and lockfile diffs called out explicitly by the author (C70/72); error-message and log-event additions reviewed as product surface (C60). Review checklist = Doc 25 gates. |
| **Comments** | Comments state constraints and invariants the code cannot express ("order matters: facts before annotations, C11") — never narration of what the next line does, never change-history. Public surfaces get doc comments with the contract, not the implementation. |
| **Markdown** | One H1 per doc; status line under the title; Mermaid for all diagrams (renders on GitHub); tables for enumerable facts, prose for reasoning; relative links only. |
| **ADRs** | One file per ADR in `docs/adr/`, numbered, append-only; status field (Proposed/Accepted/Superseded-by-N); format: Context → Decision → Alternatives → Consequences; required for: new dependency edges, new tools, schema changes, scope changes, Constitution amendments (C73). |
| **Commits** | Conventional Commits (`feat|fix|docs|test|refactor|perf|chore(scope): ...`); scope = module name; breaking changes flagged; DCO sign-off required (ADR-016). |
| **Branches** | Trunk-based: short-lived branches off `main` (`<type>/<slug>`), merge via PR + merge queue; no long-lived release branches pre-1.0; release lines (`v1.x`) only after GA and only for backported fixes. |
| **Versioning** | SemVer via changesets; the three public contracts (CLI/MCP surface, runner-sdk, store schema) each have explicit compat tests and their own compatibility notes in every release (Doc 14); pre-1.0 breaking changes allowed at minors with migration notes. |
